import pandas as pd
from google.cloud import bigquery
import google.auth # Import google.auth

# --- Configuration ---
PROJECT_ID = 'microgrid-demo'
DATASET_ID = 'microgrid_db'
TABLE_ID = 'microgrid_telemetry'
LOCATION = 'australia-southeast1' # BigQuery dataset location (region)

# Initialize BigQuery client
# Explicitly fetch credentials and pass them to the client
credentials, default_project_id = google.auth.default()
client = bigquery.Client(project=PROJECT_ID, credentials=credentials)

# --- Fetch Data from BigQuery ---
# This query fetches the latest 1000 records from the telemetry table
# You might adjust this based on your data volume and processing needs.
query = f"""
SELECT
    timestamp,
    site_id,
    battery_v,
    battery_soc,
    battery_current,
    battery_temp,
    ac_input_v,
    ac_output_v,
    ac_output_i,
    ac_input_power,
    ac_output_power,
    solar_w,
    load_w,
    inverter_state,
    inverter_temp,
    fault_code,
    power_balance_w
FROM
    `{PROJECT_ID}.{DATASET_ID}.{TABLE_ID}`
ORDER BY
    timestamp DESC
LIMIT 1000
"""

# Run the query and load results into a Pandas DataFrame
try:
    df = client.query(query).to_dataframe()
    print(f"Successfully fetched {len(df)} records from BigQuery.")
    print("
DataFrame head:")
    print(df.head())
    print("
DataFrame info:")
    df.info()

except Exception as e:
    print(f"An error occurred: {e}")

# --- Basic Feature Engineering (Example) ---
# Convert timestamp to datetime objects and set as index
df['timestamp'] = pd.to_datetime(df['timestamp'])
df = df.set_index('timestamp')

# Sort by timestamp (important for time-series operations)
df = df.sort_index()

# Example: Calculate a rolling mean for battery_soc for each site
# This is a common technique for anomaly detection (deviation from rolling average)
# For more advanced feature engineering, consider group-by operations on 'site_id'

# Ensure data is sorted by site_id and then timestamp for correct rolling calculations
df_sorted = df.sort_values(by=['site_id', 'timestamp'])

# Calculate rolling mean of battery_soc for each site
df_sorted['battery_soc_rolling_mean_5min'] = df_sorted.groupby('site_id')['battery_soc'].transform(lambda x: x.rolling('5min').mean())
df_sorted['battery_soc_deviation_5min'] = df_sorted['battery_soc'] - df_sorted['battery_soc_rolling_mean_5min']

print("
DataFrame with new features head:")
print(df_sorted.head())

## Phase 11: Vertex AI Workbench — BigQuery Connectivity Fix (Completed)

- **Issue:** Jupyter notebook on `microgrid-anomaly-detection-notebook` (Vertex AI Workbench) failed with:
  `403 request failed: the user does not have 'bigquery.readsessions.create' permission for 'projects/microgrid-demo'`
- **Root Cause 1:** `bigquerystorage.googleapis.com` (BigQuery Storage API) was not enabled. The `to_dataframe()` call in the BigQuery Python client uses the Storage API by default for fast columnar reads — this is separate from the core `bigquery.googleapis.com` API.
- **Root Cause 2:** `bigquery.readsessions.create` is NOT included in `roles/bigquery.dataEditor` or `roles/bigquery.dataViewer`. It exists only in `roles/bigquery.admin`, which is too broad to grant.
- **Fix Applied:**
    1. Enabled API: `gcloud services enable bigquerystorage.googleapis.com --project=microgrid-demo`
    2. Created custom IAM role `projects/microgrid-demo/roles/bigquery_storage_reader` with permissions:
       - `bigquery.readsessions.create`
       - `bigquery.readsessions.getData`
       - `bigquery.readsessions.update`
    3. Bound role to `microgrid-agent@microgrid-demo.iam.gserviceaccount.com`
- **Outcome:** Successfully fetched 1000 records from `microgrid_db.microgrid_telemetry` into a Pandas DataFrame. All 17 columns confirmed, zero nulls, 10-second cadence data across 11 sites validated. Rolling SOC feature engineering also confirmed working.

## Phase 23: Dashboard VM Migration — microgrid.tinylab.ai (Completed 2026-05-19)

### Objective
Move the Streamlit dashboard from the developer's local WSL laptop to the
existing `microgrid-broker` VM so it is always-on and publicly accessible
over HTTPS at `https://microgrid.tinylab.ai`.

### Decision: VM over Cloud Run
Streamlit is a stateful, long-lived WebSocket application. Cloud Run is
fragile for this workload (idle connection timeouts, session state isolation
across instances). The broker VM (e2-small) had ~700MB free RAM — sufficient
headroom to run Streamlit alongside Mosquitto, Redis, and the bridge.

### Infrastructure changes
- **GCP firewall:** `allow-http-https` rule created — ports 80 and 443 opened
  on `http-server` and `mqtt-server` network tags.
- **DNS:** `A microgrid.tinylab.ai → 34.87.254.184` added at registrar.
- **Nginx:** Installed as reverse proxy — TLS termination on 443, HTTP→HTTPS
  redirect on 80, WebSocket upgrade headers for Streamlit's `/_stcore/stream`.
- **Certbot:** Let's Encrypt certificate issued for `microgrid.tinylab.ai`;
  auto-renews via systemd timer.
- **Systemd unit:** `microgrid-dashboard.service` — Streamlit bound to
  `127.0.0.1:8501`, `MemoryMax=1500M`, `Restart=always`, runs as `amplifiedengr`.
- **Repo cloned to VM:** `/opt/microgrid` — `git pull` is now the update path.
- **Python venv:** `/opt/microgrid/.venv` with dashboard dependencies.

### New scripts committed to repo
- `scripts/deploy_dashboard_vm.sh` — orchestrates full deploy from local machine
  (firewall, IAM check, file copy, VM setup).
- `scripts/vm_setup_dashboard.sh` — runs on the VM; installs packages, clones
  repo, creates venv, enables services.
- `scripts/microgrid-dashboard.service` — systemd unit file.
- `scripts/nginx-microgrid.conf` — clean nginx config (post-certbot, no duplicate
  server blocks).

### Incident during migration: mqtt_to_pubsub.py lost
During VM setup, `sudo rm -rf /opt/microgrid` was required to clear a
pre-existing non-git directory before cloning. This deleted the original
`mqtt_to_pubsub.py` and its venv — the file had never been in the GitHub repo.

The running `mqtt-bridge` process (PID 62003, alive since 2026-03-26) survived
in memory but began failing at 11:03 UTC with:
  `OSError: Could not find a suitable TLS CA certificate bundle,
   invalid path: /opt/microgrid/venv/lib/python3.11/site-packages/certifi/cacert.pem`

**Recovery:**
1. Recreated venv at `/opt/microgrid/venv` with original packages:
   `certifi, google-cloud-pubsub, paho-mqtt, redis`.
2. Reconstructed `mqtt_to_pubsub.py` from CLAUDE.md architecture description,
   BQ schema, `sim_site.py` payload format, and Pub/Sub subscription inspection.
   Key finding: bridge only publishes to Pub/Sub — `bq-ingest-sub` (native
   Pub/Sub→BigQuery subscription) handles all BQ writes. No BQ client needed.
3. Created new Mosquitto bridge credential (`mqtt-bridge` / `bridge-pass-secret`).
4. `mqtt_to_pubsub.py` now committed to the repo — can never be lost again.
